1.3 运营常用 SQL 模板

💻 每日 GMV 看板查询
-- 按渠道的日 GMV 和订单数
SELECT 
    channel,
    DATE(order_time) as dt,
    COUNT(DISTINCT order_id) as orders,
    SUM(actual_amount) as gmv,
    SUM(actual_amount) / COUNT(DISTINCT order_id) as asp
FROM orders
WHERE order_status NOT IN ('cancelled', 'refunded')
  AND DATE(order_time) = CURDATE()
GROUP BY channel, DATE(order_time);
💻 库存预警查询
-- 周转天数 > 60 的滞销 SKU
WITH daily_sales AS (
    SELECT sku_id, 
           SUM(qty) / 30 as avg_daily_sales
    FROM order_items 
    WHERE order_time >= DATE_SUB(NOW(), INTERVAL 30 DAY)
    GROUP BY sku_id
)
SELECT i.sku_id, i.stock_qty,
       ROUND(i.stock_qty / s.avg_daily_sales, 1) as dos,
       CASE 
           WHEN i.stock_qty / s.avg_daily_sales > 90 THEN '严重积压'
           WHEN i.stock_qty / s.avg_daily_sales > 60 THEN '偏慢'
           WHEN i.stock_qty / s.avg_daily_sales < 7  THEN '断货风险'
           ELSE '健康'
       END as status
FROM inventory i
JOIN daily_sales s ON i.sku_id = s.sku_id
ORDER BY dos DESC;

4. 测验

🧠 场景题

早上一眼看板：GMV 正常，但「已发货」订单数比昨天同时段少了 40%。你的排查顺序？

A. 先查仓库是否出问题 → 再查支付成功率 → 最后查订单状态 B. 先查支付成功率 → 再查 WMS 出库速度 → 最后查物流 C. 先查订单状态分布 → 定位卡在哪一步 → 再查对应系统
❌ 可能猜对了方向，但应该是先看数据再下手查系统。
❌ 先假设了是支付或仓库问题，但还没看数据分布。
✅ 正确！先拉订单状态分布：待支付多少？已支付待发货多少？已发货多少？
如果「已支付待发货」暴涨 → WMS 出库慢
如果「待支付」暴涨 → 支付环节问题
如果「已发货」确实少了但 GMV 正常 → 可能是大促后的正常波动
原则：先看数据分布，再定位系统。
