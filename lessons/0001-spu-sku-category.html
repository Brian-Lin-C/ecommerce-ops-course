📦 Lesson 0001：电商运营的第一块砖——商品、SPU、SKU、类目

如果你只能理解电商的三个概念，那应该是：类目、SPU、SKU。
所有库存、订单、销售分析、数据看板，都建立在这三个概念之上。

1. 从一个真实问题开始

假设你在 Insta360 的商品管理后台，需要录入一款新产品：

🎬 场景

Insta360 发布了新款全景相机 X5，有两个颜色（黑/白），三个存储规格（64GB/128GB/256GB），在官网、Amazon、京东三个渠道销售。

问题：后台系统里应该创建几个「商品」？几个「SKU」？

如果你的答案是「1 个商品，18 个 SKU」或「3 个商品」——恭喜你已经踩到了电商系统设计最核心的坑：不同角色看到的「商品」不是同一个东西。

2.1 类目（Category）——商品的「书架」

类目是商品的分类体系，决定了商品在哪个货架上被用户找到。

🏪 类目层级示例（消费电子品牌）
电子数码
  └─ 摄影摄像
       └─ 运动相机          ← Insta360 主品类
       └─ 全景相机
       └─ 配件
            └─ 自拍杆
            └─ 保护壳
            └─ 镜头保护镜

为什么类目重要？

  • 前端导航和搜索依赖类目树
  • 不同类目可以有不同的属性模板（相机有「像素」「光圈」，衣服有「尺码」「面料」）
  • 销售报表按类目汇总：哪个品类贡献最多 GMV？
  • 渠道铺货规则：「配件类目只铺京东和官网，不上 Amazon」

3. SPU vs SKU —— 电商最核心的区分

SPU

Standard Product Unit
标准化产品单元

是什么：一组共用属性相同的商品集合

谁关心：平台运营、搜索、推荐

例子：「Insta360 X5」是一个 SPU

变与不变：品牌、型号、系列等共性属性不变

SKU

Stock Keeping Unit
库存量单位

是什么：库存进出计量的最小单位

谁关心：仓库、采购、财务

例子：「Insta360 X5 黑色 128GB」是一个 SKU

变与不变：颜色、容量、尺码等销售属性可变

3.1 一张表看懂关系

维度 SPU SKU
全称 Standard Product Unit Stock Keeping Unit
中文 标准化产品单元 库存量单位
粒度 粗（聚合） 细（原子）
数量关系 1 个 SPU 下面 N 个 SKU
价格 区间价（¥2,999-3,999） 精确价（¥2,999）
库存 不关心 每个 SKU 有独立库存数
条形码 无 每个 SKU 有唯一条形码

3.2 回到开头的问题

✅ 正确答案

1 个 SPU：Insta360 X5
6 个 SKU：颜色(2) × 规格(3) = 6

渠道（官网/Amazon/京东）不产生新 SKU，只是在每个渠道上架同样的 6 个 SKU。
渠道 x SKU 的铺货关系 = 6 × 3 = 18 条铺货记录，但不是 18 个 SKU。

4. 为什么会有「ID」和「货品」？

在实际中台系统中，你还会遇到两个概念：

🏷️ Item / 货品

SKU 的仓库视角版本。同一个 SKU 在不同仓库有不同库存数量，仓库管理的是「货品」而非「商品」。

商品（销售视角）：标题、图片、售价、描述 → 给消费者看
货品（仓储视角）：重量、体积、批次号、保质期 → 给仓库看

🆔 ID 体系
spu_id:     "SPU20240001"    ← 一个 SPU 一个 ID
sku_id:     "SKU20240001001"  ← 每个 SKU 一个 ID  
item_id:    "ITEM-WH01-001"   ← 仓库维度的货品 ID
channel_id: "AMAZON_US"       ← 渠道标识

运营中台的所有业务（库存、订单、销售分析）都靠这些 ID 串联。排查数据异常时，第一步永远是先确认 ID 层面的关联关系是否正确。

5. 测验

🧠 自我检测

Nike 在售一款跑鞋 Air Zoom，有 3 个颜色（黑白蓝）、5 个尺码（39-43），在天猫和京东两个渠道销售。
请问：几个 SPU？几个 SKU？

A. 2 个 SPU，15 个 SKU B. 1 个 SPU，15 个 SKU C. 1 个 SPU，30 个 SKU D. 2 个 SPU，30 个 SKU
❌ 不对。「天猫」和「京东」是渠道，不是 SPU。SPU 是产品维度的。
✅ 正确！1 个 SPU（Nike Air Zoom），15 个 SKU（3色 × 5码）。渠道不产生新 SKU，只是铺货关系。
❌ 不对。渠道不是 SKU 的乘数因子。30 = 3×5×2，但渠道是铺货维度，不是库存维度。
❌ 不对。SPU 只有一个（同一款鞋），渠道不产生新 SPU。

6. 关键要点

# Takeaway
1 电商商品体系三层：类目（组织）→ SPU（聚合）→ SKU（原子）
2 SPU 管「是什么产品」，SKU 管「哪一个具体规格」
3 库存、价格、条形码绑定在 SKU 层面，不在 SPU
4 渠道铺货 ≠ 新 SKU。渠道 × SKU = 铺货记录
5 「商品」和「货品」是两个视角——前台销售 vs 后台仓储
6 排查问题第一步：确认 ID 体系的关联关系
